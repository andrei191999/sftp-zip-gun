## ADDED Requirements

### Requirement: Output XML diagnostics
Validation errors (from local XSD, Schematron, and Helger) SHALL be written as `vscode.Diagnostic` entries keyed to the URI of the output XML document. When the output is an unsaved new tab, the extension SHALL use the `TextDocument.uri` returned by `vscode.workspace.openTextDocument`. When the output is a saved file, the extension SHALL use the file's `vscode.Uri`. Clicking a diagnostic in the Problems panel SHALL navigate the editor to the corresponding line in the output document. If the output document URI scheme does not support diagnostic attachment in the installed VS Code version, the extension SHALL fall back to reporting diagnostics at workspace level (no URI) rather than failing or throwing.

#### Scenario: Clicking error navigates to output XML line
- **WHEN** the Problems panel shows a validation error at line 42 of the output document
- **THEN** clicking the error navigates the editor to line 42 of the output XML tab

#### Scenario: Diagnostics attached to untitled output tab
- **WHEN** the output destination is `newTab` and the result is an untitled document
- **THEN** the Problems panel entry is associated with the untitled document's URI, not with any source file

### Requirement: Saxon trace file generation
When `xmlXslt.transform.enableTracing` is true, the extension SHALL invoke Saxon with the additional flags `-T -traceout:{tempFilePath}` to produce a template execution trace. The trace file SHALL be written to a temporary path and deleted after parsing. If tracing is disabled, these flags SHALL be omitted and no trace file is produced.

#### Scenario: Trace file produced when tracing enabled
- **WHEN** `enableTracing` is true and a transform runs
- **THEN** Saxon is invoked with `-T -traceout:{path}` and the trace file is written to disk

#### Scenario: No trace file when tracing disabled
- **WHEN** `enableTracing` is false
- **THEN** Saxon is invoked without `-T` or `-traceout`, and no trace file is created or read

---

### Requirement: Output-line to XSLT-line map from Saxon trace
The extension SHALL parse the Saxon `-T` trace file to build a `Map<number, vscode.Location>` mapping each output line number to the XSLT source location (file URI + line number) of the template rule that generated it. If the trace file is absent, empty, or unparseable, the map SHALL be empty (tracing degrades gracefully to no Related Information).

#### Scenario: Trace map populated from valid trace file
- **WHEN** the Saxon trace file contains entries associating output line 42 with XSLT line 118
- **THEN** `traceMap.get(42)` returns a `vscode.Location` pointing to the XSLT file at line 118

#### Scenario: Missing trace file produces empty map
- **WHEN** the trace file cannot be read (Saxon error or tracing disabled)
- **THEN** `traceMap` is empty and diagnostics are created without Related Information

---

### Requirement: Related Information links to XSLT source
For each validation diagnostic on the output document, if the trace map contains an entry for the diagnostic's line number, the extension SHALL add a `DiagnosticRelatedInformation` entry pointing to the XSLT file and line number. The related information message SHALL read "Generated by XSLT template at this location".

#### Scenario: Error with trace entry shows XSLT link
- **WHEN** output line 42 has a validation error and `traceMap.get(42)` returns an XSLT location
- **THEN** the diagnostic includes a Related Information link to the XSLT file at the mapped line, visible as a secondary location in the Problems panel

#### Scenario: Error without trace entry has no XSLT link
- **WHEN** output line 15 has a validation error and `traceMap.get(15)` returns undefined
- **THEN** the diagnostic is created without Related Information — no broken link, no error

---

### Requirement: Comment-injection fallback tracing
When `xmlXslt.transform.enableTracing` is true but the Saxon trace map is empty (i.e., the trace file was unparseable or produced no usable line mappings), the extension SHALL attempt the comment-injection fallback from `xsltTracer.ts`. The fallback MUST NOT be applied if tracing is disabled (the user has opted out entirely) or if the XSLT output method is `text` (injected XML comments would corrupt text output).

#### Scenario: Fallback used when tracing enabled but trace map empty
- **WHEN** `enableTracing` is true, the Saxon trace file yields no usable entries, and the XSLT output method is not `text`
- **THEN** the comment-injection fallback runs and produces a best-effort XSLT location mapping

#### Scenario: Fallback skipped when tracing disabled
- **WHEN** `enableTracing` is false
- **THEN** no fallback runs — the user has opted out of tracing entirely, so no Related Information is added

#### Scenario: Fallback skipped for text output method
- **WHEN** the XSLT declares `<xsl:output method="text">` (regardless of tracing setting)
- **THEN** the comment-injection fallback does not run (injected XML comments would corrupt text output); diagnostics are created without Related Information
